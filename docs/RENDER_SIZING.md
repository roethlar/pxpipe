# How pxpipe sizes rendered images

The current proxy renders only eligible exact in-place Anthropic spans. OpenAI
Chat and Responses requests — including Codex/Sol and Grok — stay byte-for-byte
unchanged. The OpenAI geometry below is retained for standalone export, offline
tests, and historical receipts; it is not an active proxy path.

Where rendering is allowed, image geometry is selected by the exact model id.
`/v1/responses` is only a wire protocol, not permission to transform a request.

## Renderer behavior and current use

A page is profile-width and variable-height. It is never padded to a square.

```text
cellWidth  = atlasWidth  + cellWBonus
cellHeight = atlasHeight + cellHBonus
width      = 2 * PAD_X + columns * cellWidth
height     = 2 * PAD_Y + usedRows * cellHeight
```

Width may shrink to the widest rendered line, capped by the profile. Height
grows with content until the profile cap, then overflow becomes another page.

| model rule | proxy use | atlas / effective cell | columns | full width | max height |
|---|---|---|---:|---:|---:|
| Claude / Anthropic | active for admitted exact spans | Spleen + Unifont, 5×8 | 312 | 1568 px | 728 px |
| `gpt-5.6-sol` | inactive; pass-through | JetBrains Mono 10 + Unifont fallback, 6×11 | 126 | 764 px | 1932 px |
| Grok | inactive; pass-through | Spleen + Unifont, effective 9×12 | 84 | 764 px | 1932 px |
| other OpenAI fallback | inactive; pass-through | Spleen + Unifont, 5×8 | 152 | 768 px | 1932 px |

Complete profile values and evidence are in
[`MODEL_RENDER_PROFILES.md`](MODEL_RENDER_PROFILES.md).

The Sol row is an offline geometry, not a successful recall claim. Historical
paid raw-image calls tested both that 6×11 row and the old 5×8/152-column
fallback: each returned 0/4 exact identifiers with four confabulations; 5×8 also
missed gist. No setting can enable either profile in the current proxy. See
[`eval/sol-profile/RESULTS.md`](../eval/sol-profile/RESULTS.md).

## Provider boundaries

Anthropic resizes images to fit both a 1568px long edge and roughly 1.15MP.
The 1568×728 Claude page stays inside both bounds, so rasterized glyphs reach
the vision encoder without the old 0.555× resample.

The dormant OpenAI profiles use portrait strips at or below the 768px short-side
floor and a 1932px height cap. GPT 5.6 Sol's 126×6+8 and Grok's 84×9+8 both
produce 764px full-width strips in offline rendering only.

## Font and Unicode

Atlases are generated at build time; runtime never loads font files. The compact
JetBrains atlas contains common code/Latin/symbol ranges. Missing codepoints
fall back to the full Spleen/Unifont atlas, preserving multilingual rendering.

Spacing, anti-aliasing, grid, color cycle, and newline-marker style are all part
of the model profile. Legacy slab/history helpers retain these values for tests;
the current proxy does not render those buckets.

## Gate parity

The active Anthropic profitability gate resolves the same profile as the
renderer and derives:

- cell width and height
- content-width shrink
- rows per page
- page count
- per-image token cost

Changing cell spacing or font therefore cannot leave active Anthropic cost
prediction on a different canvas. GPT tile/patch and Grok megapixel pricing
helpers remain for offline evaluation; OpenAI proxy requests do not invoke them.

## Why not square pages

Billing follows pixel area or patches/tiles, not visual symmetry. Padding a
portrait or short final page to a square adds cost without adding information.
The target is readable characters per billed pixel under each provider's resize
contract, not a preferred aspect ratio.

Older benchmark receipts retain their historical geometry; do not rewrite those
numbers as if they were generated by the current profile.

## Changing a profile

1. Add or re-run a reader-specific recall/style harness.
2. Measure the provider's real image-token accounting.
3. Keep the strip inside its no-resize boundary.
4. Test every active exact-span renderer, admission gate, and cache boundary
   together; do not treat a dormant slab/history helper as an active path.
5. Do not generalize one variant's result to sibling model ids.
